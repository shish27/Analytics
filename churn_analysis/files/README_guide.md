# Проект: Анализ оттока клиентов (Customer Churn Analysis)

**Цель:** определить, какие клиенты склонны уходить, найти ключевые факторы оттока
и дать бизнес-рекомендации по удержанию. Стек: SQL + Python (Pandas, scikit-learn).

**Telco Customer Churn (IBM)** — Kaggle, автор `blastchar`

Скачайте `WA_Fn-UseC_-Telco-Customer-Churn.csv` и положите в папку проекта как `telco_churn.csv`.

## 3. Шаги работы

### Шаг 1 — Загрузка данных в SQLite и первичная разведка через SQL

Зачем SQL, если данные и так в CSV: это показывает работодателю, что вы умеете
формулировать бизнес-вопросы на языке запросов, а не только через pandas.
Используем SQLite — не нужно поднимать сервер, но синтаксис переносится на Postgres/MySQL.

```python
import sqlite3
import pandas as pd

df = pd.read_csv("data/telco_churn.csv")
conn = sqlite3.connect("churn.db")
df.to_sql("customers", conn, if_exists="replace", index=False)
```

Дальше выполняем запросы из `sql/churn_analysis.sql` (см. отдельный файл) — они отвечают
на конкретные бизнес-вопросы: общий процент оттока, отток по типу контракта,
отток по методу оплаты, отток по сегментам tenure, потери в выручке от ушедших клиентов.

### Шаг 2 — Очистка данных (Python)

- `TotalCharges` в исходном файле хранится как строка и содержит пробелы у новых клиентов
  (tenure = 0) — привести к числу, обработать пропуски
- Проверить дубликаты по `customerID`
- Привести бинарные Yes/No к 1/0 для будущей модели

### Шаг 3 — EDA (разведочный анализ)

Визуализации, которые обязательно должны быть в проекте:
- Распределение оттока по `Contract` (обычно помесячный контракт даёт наибольший отток)
- Распределение оттока по `tenure` (новые клиенты уходят чаще)
- Зависимость оттока от `MonthlyCharges`
- Отток по наличию `TechSupport` / `OnlineSecurity`

### Шаг 4 — Feature engineering

- Кодирование категориальных признаков (One-Hot Encoding)
- Создание сегментов tenure: 0–12 мес / 13–24 / 25–48 / 48+
- Расчёт `CLV_proxy` = `tenure * MonthlyCharges` как прокси пожизненной ценности клиента

### Шаг 5 — Моделирование

- Train/test split (80/20), стратификация по `Churn` (датасет несбалансирован — уходит ~27%)
- Baseline: Logistic Regression (интерпретируемая модель — важно для бизнес-выводов)
- Сравнение с Random Forest
- Метрики: **не только accuracy** (при дисбалансе классов она вводит в заблуждение) —
  главное **Recall на классе Churn** (не пропустить уходящих клиентов), **ROC-AUC**, **Precision**
- Матрица ошибок (confusion matrix) — обязательно показать и прокомментировать

### Шаг 6 — Интерпретация модели

- Feature importance (Random Forest) или коэффициенты (Logistic Regression)
- Определить топ-5 факторов, влияющих на отток

### Шаг 7 — Выводы и рекомендации

Это САМАЯ важная часть для портфолио аналитика — код почти никто не будет построчно
проверять, а вот выводы читают все. Не просто "у модели ROC-AUC 0.84", а:
*"Что это значит для бизнеса и что с этим делать"*.

Шаблон структуры выводов — см. `outline_conclusions.md` ниже в этом файле.

## 4. Как писать раздел "Выводы и рекомендации" (шаблон)

```markdown
## Ключевые находки

1. Общий отток составляет ~27%. Это выше среднеотраслевого показателя для
   телеком-компаний (обычно 15–25%), что говорит о системной проблеме удержания.

2. Помесячный контракт даёт отток в разы выше, чем годовой/двухгодичный.
   Клиенты без долгосрочных обязательств уходят значительно чаще.

3. Первые 12 месяцев — критический период: львиная доля оттока приходится
   именно на новых клиентов.

4. Отсутствие дополнительных услуг (TechSupport, OnlineSecurity) коррелирует
   с более высоким оттоком — вероятно, из-за более слабой вовлечённости клиента.

5. Способ оплаты Electronic check связан с повышенным оттоком по сравнению
   с автоплатежом — возможно, маркер менее лояльного или финансово нестабильного сегмента.

## Рекомендации бизнесу

1. Предлагать переход на годовой контракт с бонусом в первые 3 месяца — именно
   там риск ухода максимален.

2. Внедрить онбординг-программу для новых клиентов (первые 12 месяцев) —
   проактивные касания снижают отток именно в этот период.

3. Стимулировать переход на автоплатёж (сокращает трение при оплате).

4. Предлагать TechSupport/OnlineSecurity бесплатно на пробный период —
   вовлечённость в доп. сервисы удерживает клиента.

## Ограничения анализа

- Датасет — срез на один момент времени, не временной ряд, поэтому нельзя
  строить прогноз оттока во времени, только профиль риска.
- Нет данных о причинах ухода (конкуренты, цена, качество связи) — модель
  предсказывает "кто", но не отвечает "почему" на уровне мотивации клиента.
```

## 5. Адаптация под retail-датасет (E-Commerce Customer Churn)
